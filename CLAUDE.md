Early / Experimental

## Project Context
Kubernetes operator that provides a declarative, typed alternative to raw Jobs/CronJobs. Users write `TaskRun` CRs composed of steps referencing `StepDefinition`/`ClusterStepDefinition` CRs. The operator resolves step types, generates Jobs/CronJobs for runner-based steps, executes API-native steps directly in the controller, and writes structured status (including logs) back to the `TaskRun`.

- **API group:** `taskrun.io`
- **API version:** `v1alpha1`
- **Module:** `github.com/davidkenelm/taskrun`
- **CRDs:** TaskRun (namespaced), StepDefinition (namespaced), ClusterStepDefinition (cluster-scoped)

## Guardrails
- Controller must never panic on bad user input — validate and write status errors
- API-native steps execute in the controller process — keep them fast and non-blocking
- Runner containers communicate via shared `emptyDir` at `/etc/step/` — never assume persistent storage
- StepDefinition resolution order: namespace-local first, then cluster-scoped
- All config must be externalizable — no hardcoded values

## Patterns
- **Kubebuilder v4** project layout with controller-runtime v0.23+
- CRD types in `api/v1alpha1/`, controllers in `internal/controller/`
- Validating webhook in `internal/webhook/v1alpha1/`
- Custom Prometheus metrics in `internal/metrics/`
- Shared `StepDefinitionSpec` type between `StepDefinition` and `ClusterStepDefinition`
- Helm chart in `charts/taskrun/` with built-in `ClusterStepDefinition` CRs
